# Release

`devimg` releases are crates.io packages plus GitHub Releases built from version tags. The tag must match the workspace package version.

Source builds require Rust 1.85 or newer. The GitHub release workflow uses the current stable toolchain.

## Pre-Publish Checks

Run the same checks locally before publishing:

```bash
cargo +1.85.1 fmt --all -- --check
cargo +1.85.1 clippy --all-targets --all-features -- -D warnings
cargo +1.85.1 test --all
scripts/security-checks.sh
cargo +1.85.1 publish --dry-run -p devimg-core
cargo +1.85.1 package -p devimg --no-verify --list
```

For workspace dependencies, the CLI crate uses `devimg-core = { version = "...", path = "../devimg-core" }` so local development uses the workspace copy and published crates use the registry version.
`cargo publish --dry-run -p devimg` can only pass after the matching `devimg-core` version exists in the crates.io index.

## Publish To crates.io

Publish the library first, then the CLI:

```bash
cargo login <crates.io-api-token>
cargo +1.85.1 publish -p devimg-core
cargo +1.85.1 publish --dry-run -p devimg
cargo +1.85.1 publish -p devimg
```

`cargo install devimg` installs the CLI binary named `devimg`.

## Publish GitHub Release

For version `0.1.14`:

```bash
git tag v0.1.14
git push origin v0.1.14
```

The `Release` workflow builds and publishes:

- `devimg-linux-x86_64.tar.gz`
- `devimg-darwin-x86_64.tar.gz`
- `devimg-darwin-aarch64.tar.gz`
- `devimg-windows-x86_64.tar.gz`

Each archive has a matching `.sha256` checksum file.

You can also rerun the workflow manually for an existing tag from GitHub Actions by passing the tag input.

## Install From crates.io

```bash
cargo install devimg
devimg --help
```

Source installs require Rust 1.85 or newer. If the active default toolchain is older, use `rustup update stable` or run `cargo +1.85.1 install devimg`.

## Install From GitHub

Download the matching archive from the release page, verify its checksum, extract it, and place `devimg` on `PATH`.

Example for Linux x86_64:

```bash
curl -fsSLO https://github.com/cleissonom/devimg/releases/download/v0.1.14/devimg-linux-x86_64.tar.gz
curl -fsSLO https://github.com/cleissonom/devimg/releases/download/v0.1.14/devimg-linux-x86_64.tar.gz.sha256
sha256sum -c devimg-linux-x86_64.tar.gz.sha256
tar -xzf devimg-linux-x86_64.tar.gz
./devimg --help
```

## Cargo Install

Install directly from the Git tag:

```bash
cargo install --git https://github.com/cleissonom/devimg --tag v0.1.14 devimg
```

For local development:

```bash
cargo install --path crates/devimg-cli
```

## GitHub Action

Consumer workflows should pin the Action to the release tag:

```yaml
- uses: cleissonom/devimg/action@v0.1.14
  with:
    config: devimg.toml
    mode: check
```

The Action downloads the release archive matching the runner OS and architecture unless `binary-path` is supplied.
It verifies the matching `.sha256` file before extracting the binary.

When a project commits a manifest helper generated by `devimg manifest export`, use the Action export drift inputs documented in `docs/action.md`.

## Repository Visibility Notes

Public Action usage requires the repository to be public, because workflow users reference a subdirectory Action as:

```yaml
- uses: cleissonom/devimg/action@v0.1.14
```

This is intentionally not a GitHub Marketplace release yet. Marketplace packaging can be revisited later with a separate root-level Action repository if needed.

## Private Repository Notes

GitHub Releases work in private repositories, but release assets are only visible to users and workflows with read access to that repository.

For GitHub Actions consumers, use public repository settings only when broad public Action usage is intentionally chosen. For scoped private use, configure the repository's Actions access settings so allowed private repositories can use the private Action, and provide a token with access to the private release assets.
