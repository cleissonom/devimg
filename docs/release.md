# Release

`devimg` releases are crates.io packages plus GitHub Releases built from version tags. The tag must match the workspace package version.

Source builds require Rust 1.88 or newer. The GitHub release workflow uses the current stable toolchain.

## Pre-Publish Checks

Run these checks locally before publishing:

```bash
cargo +1.88.0 fmt --all -- --check
cargo +1.88.0 clippy --all-targets --all-features -- -D warnings
cargo +1.88.0 test --all
scripts/security-checks.sh
cargo +1.88.0 publish --dry-run -p devimg-core
cargo +1.88.0 package -p devimg --no-verify --list
```

The CLI crate uses a workspace path dependency with a registry version:

```toml
devimg-core = { version = "...", path = "../devimg-core" }
```

Local development uses the workspace copy. Published crates use the registry version. Because of that, `cargo publish --dry-run -p devimg` can only pass after the matching `devimg-core` version is available in the crates.io index.

## Publish To crates.io

Publish the library first, then the CLI:

```bash
cargo login <crates.io-api-token>
cargo +1.88.0 publish -p devimg-core
cargo +1.88.0 publish --dry-run -p devimg
cargo +1.88.0 publish -p devimg
```

After publishing, verify the public install path:

```bash
cargo install devimg --force
devimg --version
devimg --help
```

Crates are effectively permanent. If a bad version is published, yank it with `cargo yank`; do not treat yanking as secret removal.

## Publish GitHub Release

Tag the same commit after crates.io publish succeeds:

```bash
VERSION=0.2.7
git tag "v${VERSION}"
git push origin "v${VERSION}"
```

The `Release` workflow builds and publishes:

- `devimg-linux-x86_64.tar.gz`
- `devimg-darwin-x86_64.tar.gz`
- `devimg-darwin-aarch64.tar.gz`
- `devimg-windows-x86_64.tar.gz`

Each archive has a matching `.sha256` checksum file. Wait for the workflow to pass, then download one archive, verify its checksum, and run `devimg --version`.

You can also rerun the workflow manually for an existing tag from GitHub Actions by passing the tag input.

## Install From GitHub

Download the matching archive from the release page, verify its checksum, extract it, and place `devimg` on `PATH`.

Example for Linux x86_64:

```bash
VERSION=v0.2.7
curl -fsSLO "https://github.com/cleissonom/devimg/releases/download/${VERSION}/devimg-linux-x86_64.tar.gz"
curl -fsSLO "https://github.com/cleissonom/devimg/releases/download/${VERSION}/devimg-linux-x86_64.tar.gz.sha256"
sha256sum -c devimg-linux-x86_64.tar.gz.sha256
tar -xzf devimg-linux-x86_64.tar.gz
./devimg --help
```

## Cargo Install From Git

Install directly from a Git tag:

```bash
cargo install --git https://github.com/cleissonom/devimg --tag v0.2.7 devimg
```

For local development:

```bash
cargo install --path crates/devimg-cli
```

## GitHub Action

Consumer workflows should pin the Action to a release tag:

```yaml
- uses: cleissonom/devimg/action@v0.2.7
  with:
    mode: check
```

The Action downloads the release archive matching the runner OS and architecture unless `binary-path` is supplied. It verifies the matching `.sha256` file before extracting the binary.

When a project commits a manifest helper generated by `devimg manifest export`, use the Action export drift inputs documented in `docs/action.md`.

The Action is not a GitHub Marketplace listing. Marketplace packaging can be revisited later with a separate root-level Action repository if needed.
