# DevImg v0.1.13 Plan: Framework Consumption + Public Story Pack

## Objective

Make DevImg easier to understand, review, and adopt in real frontend repositories without changing its core product shape.

The release should explain DevImg's public story, polish examples and review artifacts, and improve diagnostics for apps that consume generated images.

## Product Thesis

DevImg is a repository-owned image pipeline for frontend projects:

- deterministic generated variants;
- checked-in manifests and app helper exports;
- CI-enforced freshness and budgets;
- human-reviewable reports and visual artifacts;
- AI-agent-friendly outputs and file ownership rules.

DevImg does not try to replace image engines, framework image components, or hosted transformation services. It coordinates them from the repository and CI layer.

## Scope

### 1. README And Public Story Polish

- Rework the README introduction around the problem DevImg solves.
- Add a concise "Where DevImg fits" section comparing it to:
  - image engines such as `sharp`;
  - compression tools;
  - framework image components;
  - hosted/on-the-fly image services.
- Make the first-screen usage story clearer:
  - configure;
  - optimize;
  - export manifest helpers;
  - check in CI;
  - inspect visual review artifacts.
- Add a small "What changes in a PR?" section that explains generated variants, manifests, helper exports, reports, and review artifacts.
- Keep install/distribution guidance scoped to GitHub Release binaries and source builds until public distribution is intentionally chosen.

### 2. Examples And Review Artifact Polish

- Tighten `examples/README.md` so each example has a specific adoption lesson.
- Add a short `examples/dogfood/README.md` section showing:
  - content-hash filenames;
  - `cover` vs `contain`;
  - TypeScript helper export;
  - static review artifact.
- Ensure the dogfood review artifact is generated by CI/local commands and clearly treated as reproducible output.
- Add README links to the review artifact workflow rather than committing screenshots as source-of-truth assets.
- If screenshots are added later, they should be generated or refreshed through an explicit documented command.

### 3. Framework Consumption Diagnostics V2

Improve `devimg doctor` diagnostics for projects that consume generated images from app code.

Potential checks:

- Detect Next.js projects that use generated images under `public/`.
- Detect whether a checked-in manifest export/helper path is configured through `doctor --export-output`.
- Detect common helper import paths such as `lib/devimg.generated.ts` and `lib/devimg.ts`.
- Warn when content-hash filenames are enabled but no helper export is checked or discoverable.
- Explain the three intended consumption modes:
  - plain `img` / `picture` using exported variant URLs;
  - framework `Image` with `unoptimized` when DevImg owns sizing/quality;
  - framework image optimization intentionally layered on top of DevImg-generated source variants.
- Keep diagnostics advisory and read-only.

### 4. cleisson.com Project Page Polish

Update the DevImg project entry in `cleisson.com` after the repo work is stable.

Recommended content:

- Problem statement: generated frontend images often become stale, oversized, or manually copied.
- Solution: Rust CLI + manifest + CI check + GitHub Release binary workflow.
- Dogfood proof: `cleisson.com` uses DevImg for project card/banner images.
- Include a compact workflow list:
  - `devimg optimize`;
  - `devimg manifest export`;
  - `devimg check --fail-on-warning`;
  - static review artifact.
- Avoid promising hosted services, automatic PR commits, or AI automation before those features exist.

### 5. Share Package

Prepare a small public sharing kit after the README/project page is polished:

- short LinkedIn post draft;
- longer technical note outline;
- screenshot/review-artifact capture guidance;
- concise repo description and topic suggestions.

## Non-Goals

- Do not publish crates.io packages in v0.1.13.
- Do not add a generic installer.
- Do not make the GitHub Action a broad public distribution commitment.
- Do not add hosted storage, dashboards, or SaaS behavior.
- Do not add AI-powered behavior in v0.1.13; keep that in the v0.2 plan.
- Do not auto-edit consumer app code from `doctor`.

## Done Criteria

- README tells a clear public story in the first few sections.
- Examples explain what each fixture proves.
- Review artifact workflow is documented for local and CI use.
- `devimg doctor` has stronger framework-consumption diagnostics with tests.
- `cleisson.com/projects/devimg` is updated after the DevImg repo changes are verified.
- LinkedIn/share draft exists as a reviewable document.
- Verification covers:
  - `cargo +1.85.1 fmt --all -- --check`;
  - `cargo +1.85.1 test --all`;
  - `cargo +1.85.1 clippy --all-targets --all-features -- -D warnings`;
  - dogfood optimize/check/export/doctor/review flow;
  - YAML/action validation;
  - diff whitespace check.

## Suggested Implementation Order

1. Polish README and example docs.
2. Add framework-consumption diagnostics and tests.
3. Regenerate dogfood review/export artifacts if behavior requires it.
4. Update `cleisson.com/projects/devimg`.
5. Prepare the share package.
6. Stabilize and release `v0.1.13`.
